## 1. 김치 프리미엄이란?

### 쉽게 말하면

한국에서 비트코인이 해외보다 **비싸게** 팔리는 현상이다.

```
해외 비트코인 가격: $50,000 × 환율 1,350원 = 67,500,000원
한국 비트코인 가격:                              69,525,000원  ← 3% 더 비쌈!
```

이 차이가 **김치 프리미엄(KP)**이다. 보통 1~5% 사이를 오간다.

### 왜 발생하나?

| 원인 | 설명 |
|------|------|
| **자본 이동 제한** | 한국 외환법상 원화↔달러 자유 이동이 어려움 |
| **수요 폭발** | 한국 투자자 매수 열기가 높으면 가격 상승 |
| **차익거래 장벽** | 해외에서 싸게 사서 한국에서 비싸게 팔기가 현실적으로 어려움 |
| **시장 심리** | 공포/탐욕이 한국 시장에 더 증폭되어 반영 |

### 프리미엄 변동 패턴

```
프리미엄 높음 (3%+)  ← 한국 매수 열기 과열
       ↕
프리미엄 보통 (1~2%)  ← 평상시
       ↕
프리미엄 낮음 (0% 이하) ← 한국 투매, 역프리미엄
```

김치랑은 이 **프리미엄이 올라갔다가 내려오는 사이클**을 이용한다.

## 2. 김치랑은 어떻게 돈을 버나?

### 핵심 원리: 델타 뉴트럴 차익거래

"델타 뉴트럴"이란 **비트코인 가격 자체의 등락에 영향을 받지 않는** 전략이다.

#### Step 1: 진입 (프리미엄이 높을 때)

```
┌─────────────────────┐    ┌─────────────────────┐
│      업비트          │    │    바이낸스 선물      │
│                     │    │                     │
│   BTC 매수 (롱)     │    │   BTC 매도 (숏)      │
│   67,500,000원      │    │   $50,000           │
│                     │    │                     │
│   "한국에서 비싸게   │    │   "해외에서 싸게      │
│    사는 비트코인"    │    │    파는 비트코인"     │
└─────────────────────┘    └─────────────────────┘
         ↑                          ↑
         └──── 동시에 실행! ─────────┘

이 시점 김치 프리미엄 = 3.0%
```

#### Step 2: 청산 (프리미엄이 줄어들 때)

```
┌─────────────────────┐    ┌─────────────────────┐
│      업비트          │    │    바이낸스 선물      │
│                     │    │                     │
│   BTC 매도          │    │   BTC 매수 (숏 청산)  │
│   67,400,000원      │    │   $50,100           │
│                     │    │                     │
│   "한국에서 팔기"    │    │   "해외에서 되사기"   │
└─────────────────────┘    └─────────────────────┘

이 시점 김치 프리미엄 = 0.5%
```

#### 수익 계산

```
프리미엄 캡처:  3.0% - 0.5% = 2.5% (총 수익)

수수료 차감:
  업비트 매수 수수료:    0.05%
  업비트 매도 수수료:    0.05%
  바이낸스 매도 수수료:  0.04%
  바이낸스 매수 수수료:  0.04%
  ─────────────────────
  총 수수료:            0.18%

순수익 = 2.5% - 0.18% = 2.32%
```

### 왜 비트코인 가격에 영향을 안 받나?

비트코인이 오르든 내리든 상관없다:

```
비트코인 10% 상승 시:
  업비트 롱:    +10% 수익
  바이낸스 숏:  -10% 손실
  합계:          0% (상쇄)
  ─ 프리미엄 차이만 남음 ─

비트코인 10% 하락 시:
  업비트 롱:    -10% 손실
  바이낸스 숏:  +10% 수익
  합계:          0% (상쇄)
  ─ 프리미엄 차이만 남음 ─
```

이것이 **"델타 뉴트럴"**의 핵심이다. 양쪽이 서로 상쇄하므로 가격 변동 리스크가 없다.

```
  5. 손절 확인: mid_kp ≥ 8%면 강제 청산
```

## 4. 웹소켓 데이터 처리

### 웹소켓이란?

일반 API는 "물어봐야 답한다" (요청→응답). 웹소켓은 **"한 번 연결하면 계속 데이터가 흘러온다"**.

```
일반 API (HTTP):
  봇 → "가격 알려줘" → 거래소 → "67,500,000원"
  봇 → "가격 알려줘" → 거래소 → "67,510,000원"
  봇 → "가격 알려줘" → 거래소 → "67,490,000원"
  (매번 새로 물어봄 → 느림, API 제한에 걸림)

웹소켓 (WS):
  봇 ←→ 거래소 (연결 유지)
  거래소 → "67,500,000원"
  거래소 → "67,510,000원"   ← 알아서 계속 보내줌
  거래소 → "67,490,000원"
  (실시간, 지연 최소)
```

### 3개의 동시 웹소켓 채널

김치랑은 **3개의 데이터 소스**를 동시에 수신한다:

#### 채널 1: 업비트 웹소켓

```python
# 연결: wss://api.upbit.com/websocket/v1
# 구독: orderbook(호가) + ticker(시세)
# 포맷: SIMPLE (압축 JSON, 트래픽 절약)

수신 데이터:
  호가 → upbit_bid (매수1호가), upbit_ask (매도1호가), 수량
  시세 → 현재가, 거래량, 전일대비

속도: 초당 10+ 메시지
```

#### 채널 2: 바이낸스 웹소켓

```python
# 연결: wss://fstream.binance.com/stream
# 구독: bookTicker + 24hrTicker + markPrice (결합 스트림)

수신 데이터:
  bookTicker   → binance_bid, binance_ask, 수량
  24hrTicker   → 24시간 변동률, 거래량
  markPrice    → 마크가격, 펀딩비율, 다음 펀딩 시간

속도: 초당 20+ 메시지
```

#### 채널 3: 환율 (HTTP 폴링)

```python
# 1차: exchangerate-api.com (60초 간격)
# 2차 (폴백): 업비트 USDT/KRW 호가

수신 데이터:
  fx_rate → 현재 USD/KRW 환율 (예: 1,350.50)

속도: 60초에 1회 (환율은 급변하지 않음)
```

### 데이터 합류: MarketState

3개 채널의 데이터가 하나의 공유 객체에 모인다:

```python
MarketState:
  upbit_bid:        67,490,000    # 업비트 매수1호가
  upbit_ask:        67,510,000    # 업비트 매도1호가
  upbit_bid_qty:    0.5           # 매수 잔량
  upbit_ask_qty:    0.3           # 매도 잔량
  binance_bid:      49,998.50     # 바이낸스 매수1호가 (USDT)
  binance_ask:      50,001.50     # 바이낸스 매도1호가 (USDT)
  binance_funding:  0.0001        # 펀딩비율 (8시간마다)
  fx_rate:          1,350.50      # USD/KRW 환율
  is_ready:         True          # 모든 데이터 수신 완료?
  last_update:      1710326400.5  # 마지막 업데이트 시간
```

### 왜 asyncio인가?

김치랑은 **스레드 없이** asyncio만 사용한다:

```
스레드 방식 (안 씀):
  Thread 1: 업비트 수신
  Thread 2: 바이낸스 수신     → 데이터 충돌 가능, 락 필요, 디버깅 어려움
  Thread 3: 환율 수신

asyncio 방식 (김치랑):
  단일 이벤트 루프:
    Task 1: 업비트 수신
    Task 2: 바이낸스 수신     → 동시 실행이지만 충돌 없음, 안전
    Task 3: 환율 수신
```

asyncio는 **"한 번에 하나만 실행하지만 대기 시간에 다른 일을 한다"**. 네트워크 I/O가 대부분인 트레이딩 봇에 이상적이다.

## 5. 김치 프리미엄 계산 공식

### 3가지 프리미엄 값

김치랑은 목적에 따라 **3가지 프리미엄**을 동시에 계산한다:

#### Entry KP (진입용 — 가장 보수적)

```
entry_kp = (업비트_매도호가 / (바이낸스_매수호가 × 환율) - 1) × 100

예시:
  업비트 매도호가 = 67,510,000원 (우리가 사야 할 가격 = 비쌈)
  바이낸스 매수호가 = 49,998원 (우리가 팔아야 할 가격 = 쌈)
  환율 = 1,350원

  entry_kp = (67,510,000 / (49,998 × 1,350) - 1) × 100
           = (67,510,000 / 67,497,300 - 1) × 100
           = 0.019% ← 스프레드 포함하면 실제 프리미엄 낮음
```

**용도**: "지금 진입하면 실제로 얼마나 이득인가?" → 비싸게 사고 싸게 파는 최악 시나리오 반영

#### Exit KP (청산용 — 가장 낙관적)

```
exit_kp = (업비트_매수호가 / (바이낸스_매도호가 × 환율) - 1) × 100

예시:
  업비트 매수호가 = 67,490,000원 (우리가 팔 수 있는 가격)
  바이낸스 매도호가 = 50,001원 (우리가 되살 가격)
  환율 = 1,350원

  exit_kp = (67,490,000 / (50,001 × 1,350) - 1) × 100
          = 약 -0.01%
```

**용도**: "지금 청산하면 프리미엄이 충분히 줄었나?" → exit_kp ≤ 0.5%면 청산 실행

#### Mid KP (모니터링 — 중간값)

```
mid_kp = ((업비트_bid+ask)/2 / ((바이낸스_bid+ask)/2 × 환율) - 1) × 100
```

**용도**: RL 모델 입력, 로깅, 텔레그램 보고, 손절 판단

### 임계값 (기본 설정)

| 상황 | 조건 | 설명 |
|------|------|------|
| **진입** | entry_kp ≥ 3.0% | 프리미엄이 충분히 높아 수익 가능 |
| **청산** | exit_kp ≤ 0.5% | 프리미엄이 충분히 줄어 차익 실현 |
| **손절** | mid_kp ≥ 8.0% | 역프리미엄 폭등, 손실 제한 |

## 6. 동시 주문 실행

### 왜 "동시"가 중요한가?

```
만약 순차적으로 실행하면:
  10:00:00 업비트 매수 완료
  10:00:01 바이낸스 숏... 이 사이 가격 변동!
  → 의도한 프리미엄과 다른 가격에 체결 = 슬리피지 손실

동시 실행:
  10:00:00 업비트 매수 + 바이낸스 숏 (동시)
  → 프리미엄 그대로 캡처
```

### 실행 코드 흐름

```python
# 진입 (asyncio.gather로 동시 실행)
async def enter(snapshot):
    upbit_task = upbit_market_buy(qty_krw)      # 업비트 시장가 매수
    binance_task = binance_market_sell(qty_btc)  # 바이낸스 시장가 매도(숏 오픈)

    upbit_result, binance_result = await asyncio.gather(
        upbit_task, binance_task,
        return_exceptions=True  # 하나가 실패해도 다른 것 결과 받음
    )
    # 타임아웃: 5초
```

### 편측 실패 처리 (Leg Risk)

동시 주문인데 한쪽만 성공하면? → **응급 해제(Emergency Unwind)**

```
시나리오 1: 업비트 성공, 바이낸스 실패
  → 업비트에서 방금 산 BTC를 즉시 되팔기 (시장가 매도)
  → 소액 손실 (스프레드만큼) 감수, 포지션 정리

시나리오 2: 바이낸스 성공, 업비트 실패
  → 바이낸스에서 방금 연 숏을 즉시 커버 (시장가 매수)
  → 소액 손실 감수, 포지션 정리

시나리오 3: 둘 다 실패
  → 로그 기록, 포지션 변경 없음, 다음 기회 대기

시나리오 4: 둘 다 성공 ✓
  → 포지션 상태 저장, 청산 시그널 대기
```

### 주문 전 안전 검증 (5단계)

```
1. EMERGENCY_STOP 확인        → 긴급 정지 상태면 거부
2. 기존 포지션 확인            → 이미 포지션 있으면 중복 진입 거부
3. 일일 거래 횟수 확인         → 20회 초과면 거부
4. 최소 거래 간격 확인         → 마지막 거래 후 60초 미경과면 거부
5. 슬리피지 추정              → 0.3% 초과 예상이면 거부
```

## 7. 포지션 청산(Exit)

### 청산이란?

진입할 때 열었던 **양쪽 포지션을 동시에 닫는 것**이다.

```
진입 상태:
  업비트:   BTC 0.00015 보유 (롱)
  바이낸스: BTC 0.00015 숏 (미결제)

청산:
  업비트:   BTC 0.00015 매도 → 원화 회수
  바이낸스: BTC 0.00015 매수 → 숏 포지션 종료
```

### 3가지 청산 트리거

#### 1. 정상 청산 (프리미엄 수렴)

```
조건: exit_kp ≤ 0.5% (프리미엄이 충분히 줄었을 때)
결과: 프리미엄 차이만큼 순수익

예시:
  진입 시 entry_kp = 3.2%
  청산 시 exit_kp  = 0.4%
  총 수익 = 3.2% - 0.4% - 0.18%(수수료) = 2.62%
```

#### 2. 손절 청산 (역프리미엄 폭등)

```
조건: mid_kp ≥ 8.0% (프리미엄이 오히려 더 벌어질 때)
결과: 손실 제한 (프리미엄이 더 올라가기 전에 탈출)

이런 상황은 극히 드물지만 (시장 패닉),
방치하면 손실이 무한히 커질 수 있으므로 반드시 필요
```

#### 3. RL 판단 청산

```
조건: RL 모델이 EXIT(2) 액션 출력
결과: 모델이 학습한 최적 타이밍에 청산

RL은 단순 임계값보다 더 복잡한 패턴을 포착할 수 있다:
  - 프리미엄이 0.8%인데 더 내려갈 것 같으면 → 기다림
  - 프리미엄이 1.2%인데 반등 징조 보이면 → 미리 청산
```

### 청산 후 수익 계산

```python
# 수익률 계산 로직
pnl_pct = entry_kp - exit_kp - total_fees

# 원화 수익 계산
profit_krw = trade_amount * pnl_pct / 100

# 예: 10만원 투자, 2.5% 수익
profit = 100,000 × 2.5 / 100 = 2,500원
```

## 8. RL 강화학습 모델

### 강화학습이 뭔가? (쉬운 설명)

```
전통적 프로그래밍:
  "프리미엄 3% 이상이면 사라" ← 사람이 규칙을 정함

강화학습:
  "과거 2년 데이터를 줄 테니, 스스로 최적의 매매 타이밍을 찾아라"
  → AI가 수만 번 시뮬레이션하며 학습
  → 사람이 생각 못한 패턴도 포착
```

### 상태 벡터 (AI가 보는 세상)

RL 모델은 매 순간 **12개의 숫자**를 입력받아 판단한다:

```
[0]  mid_kp / 10          현재 프리미엄 (0.3 = 3%)
[1]  entry_kp / 10        진입 프리미엄
[2]  exit_kp / 10         청산 프리미엄
[3]  kp_ma_short / 10     단기 이동평균 (최근 추세)
[4]  kp_ma_long / 10      장기 이동평균 (큰 추세)
[5]  kp_z_score / 3       평균에서 얼마나 벗어났나 (표준편차 단위)
[6]  kp_velocity / 1      프리미엄 변화 속도 (분당 %)
[7]  spread_cost / 1      스프레드 비용 (거래 마찰)
[8]  funding_rate × 100   펀딩비율 (숏 보유 비용/수익)
[9]  volume_change         업비트 거래량 변화
[10] volume_change         바이낸스 거래량 변화
[11] position_state        현재 포지션 (0=없음, 1=보유중)
```

### 액션 공간 (AI가 할 수 있는 행동)

```
0 = HOLD  → 아무것도 안 함 (관망)
1 = ENTER → 진입 (업비트 매수 + 바이낸스 숏)
2 = EXIT  → 청산 (업비트 매도 + 바이낸스 숏 커버)
```

### PPO vs DQN: 두 가지 학습 방법

| 특성 | PPO (Policy Gradient) | DQN (Deep Q-Network) |
|------|----------------------|---------------------|
| **학습 방식** | "어떤 행동이 좋은지" 직접 학습 | "각 행동의 가치가 얼마인지" 학습 |
| **비유** | 직감적 트레이더 (감으로 판단) | 분석적 트레이더 (수치로 판단) |
| **장점** | 부드러운 정책, 안정적 | 극단적 상황(폭등/폭락) 잘 대응 |
| **단점** | 희귀 이벤트 놓칠 수 있음 | 학습 불안정할 수 있음 |
| **학습 데이터** | 90일 | 730일 (2년) |
| **학습 횟수** | 50,000 스텝 | 500,000 스텝 |
| **핵심 기법** | — | 경험 리플레이 (과거 경험 재학습) |

### 앙상블 (합의제)

두 모델을 **동시에** 사용하여 안전성을 높인다:

```
PPO 의견: "진입하자" (ENTER)
DQN 의견: "진입하자" (ENTER)
→ 합의 ✓ → ENTER 실행

PPO 의견: "진입하자" (ENTER)
DQN 의견: "기다리자" (HOLD)
→ 의견 불일치 → HOLD (보수적 선택)
```

**규칙**: 두 모델이 동의할 때만 행동. 의견이 다르면 **항상 관망**. → 불필요한 매매를 줄임.

### 보상 설계 (AI가 학습하는 기준)

```
좋은 행동 (양의 보상):
  ✓ 프리미엄 높을 때 진입하여 낮을 때 청산 → +수익률만큼

나쁜 행동 (음의 보상):
  ✗ 4스텝 이내 재진입 (과매매)         → -0.2
  ✗ 이동평균 아래에서 진입 (역추세)     → -0.05
  ✗ 불필요한 청산 (수익 없이 나감)      → -0.02
  ✗ 48시간 이상 포지션 보유 (자본 묶임)  → -0.003/스텝
```

### 폴백: RL 모델 없을 때

RL 모델이 없거나 로드 실패 시 **규칙 기반**으로 전환:

```
진입: entry_kp ≥ 3.0%
청산: exit_kp ≤ 0.5%
손절: mid_kp ≥ 8.0%
```

```
  ├── 데이터 수신: 비동기 (백그라운드 상시)

  ├── RL 추론: ~10ms
  ├── 주문 판단: ~1ms
```

```
업비트 WS:    초당 10+ 메시지
```

### 왜 속도가 중요한가?

```
  → 또는 다른 봇에게 기회 뺏김

초 단위 차이가 수익률을 결정한다.
```

## 12. 업비트 vs 바이낸스 데이터 처리 비교

### 왜 바이낸스가 더 빠른가?

```
업비트:
  - 서버: 한국 (가까움 ✓)
  - 거래량: BTC/KRW 중심 (상대적 소규모)
  - WS 포맷: 커스텀 JSON
  - 트래픽: 한국 투자자 집중 시 혼잡

바이낸스:
  - 서버: 글로벌 분산 (AWS 다중 리전)
  - 거래량: 세계 최대 (인프라 투자↑)
  - WS 포맷: 최적화된 바이너리 + JSON
  - 트래픽: 글로벌 분산 (단일 지역 혼잡 적음)
```

### 웹소켓 포맷 차이

#### 업비트 WS 메시지

```json
{
  "ty": "orderbook",
  "cd": "KRW-BTC",
  "obu": [
    {"ap": 67510000, "as": 0.3, "bp": 67490000, "bs": 0.5}
  ],
  "tas": 5.2,
  "tbs": 4.8,
  "st": "REALTIME"
}
```

- `ty`: type (orderbook/ticker)
- `cd`: code (마켓 코드)
- `obu`: orderbook units (호가 목록)
- `ap/as`: ask price/size, `bp/bs`: bid price/size
- SIMPLE 포맷으로 키 이름 축약 → 트래픽 절약

#### 바이낸스 WS 메시지

```json
{
  "stream": "btcusdt@bookTicker",
  "data": {
    "s": "BTCUSDT",
    "b": "50001.50",    // best bid
    "B": "2.500",       // bid qty
    "a": "50002.00",    // best ask
    "A": "1.800"        // ask qty
  }
}
```

- 결합 스트림으로 `bookTicker + 24hrTicker + markPrice` 한 번에 수신
- 문자열 숫자 (float 변환 필요)

### 동시 주문 시 시간차 문제

```
이상적:    업비트 체결 ─────┐
                            ├─ 동시 (0ms 차이)
           바이낸스 체결 ────┘

현실:      업비트 체결 ──────── 200ms ──────── 바이낸스 체결
           (업비트가 살짝 느림)

김치랑 대응:
  - asyncio.gather()로 동시 발송 (네트워크 출발은 동시)
  - 5초 타임아웃 (한쪽이 너무 늦으면 취소)
  - 슬리피지 0.3% 이내 허용
  - 초과 시 편측 실패 처리 → 자동 해제
```

### 데이터 신선도 관리

```
업비트 데이터가 3초 전 것이고
바이낸스 데이터가 방금 것이면?
→ 프리미엄 계산이 부정확!

김치랑의 해결:
  - 각 데이터에 타임스탬프 기록
  - 가장 오래된 데이터가 10초 이상이면 → 매매 중단
  - "데이터가 신선하지 않습니다" 로그 출력
  - 데이터 갱신될 때까지 대기
```

## 14. 자주 묻는 질문

### Q: 비트코인이 폭락하면 손해 아닌가?

**아니다.** 델타 뉴트럴이므로 비트코인 가격 자체는 상관없다.
- 업비트 롱 -10% + 바이낸스 숏 +10% = 0%
- 수익은 **프리미엄 차이에서만** 발생한다.

### Q: 그럼 뭘 때 손해를 보나?

**역프리미엄 확대** 시 손해를 본다.
- 진입 후 프리미엄이 줄기를 기대했는데 **오히려 더 벌어지면** 손실
- 이를 방지하기 위해 8% 손절선이 있다

### Q: DRY_RUN이 뭔가?

시뮬레이션 모드이다. 실제 돈이 오가지 않는다.
- `true`: 모든 분석+판단은 실행하되, 주문만 안 넣음
- `false`: 진짜 매매 실행
- **반드시 DRY_RUN=true로 충분히 테스트 후 전환할 것**

### Q: 펀딩비로도 돈을 버나?

**그렇다.** 바이낸스 숏 포지션은 펀딩비가 양수일 때 추가 수익이 발생한다.
- 평균 하루 0.03% 추가 수익
- 한 달이면 약 0.9% — 프리미엄 차익에 더해지는 보너스

### Q: 레버리지를 올리면 수익이 더 나나?

**이론적으로 그렇지만 비추천한다.**
- 1배: 사실상 청산 불가 (안전)
- 5배: 20% 역행 시 청산 → 돈 날림
- 김치랑은 소액 차익을 안전하게 반복하는 전략이므로 1배 권장

### Q: 봇이 꺼지면 포지션은?

**파일로 저장**되어 있다. 재시작하면 자동 복구된다.
- 저장 위치: `data/kimchirang_state.json`
- 진입가, 수량, 시간 등 모두 기록
- 봇 재시작 시 이전 포지션부터 이어서 관리

### Q: 환율이 급변하면?

60초마다 환율을 갱신한다. 급변 시:
- 프리미엄 계산에 반영 → 자연스럽게 조정
- 환율 소스 장애 시 업비트 USDT/KRW 시세로 폴백

### Q: 동시 주문인데 한쪽만 되면?

**자동 해제**된다.
- 성공한 쪽을 즉시 되돌리고 (반대 주문)
- 소액 손실만 발생 (스프레드)
- 포지션은 정리 상태 유지
